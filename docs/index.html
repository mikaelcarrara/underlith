WHAT IS UNDERLITH?

Design decisions as
infrastructure.

Underlith is a design token system built on CSS Variables, providing a stable, framework-agnostic foundation beneath the UI. It defines design decisions as constraints — not implementations.

Unlike UI libraries that couple design to components, Underlith defines only primitives: colors, spacing, typography, and core decisions expressed as semantic, versionable tokens.

This separation preserves design intent across platforms and implementation layers without rewriting the design system as stacks evolve.

WHY UNDERLITH EXISTS

Design systems shouldn't
break when stacks change.

Most design systems are framework-dependent — when the stack changes, they must be rebuilt.
Underlith separates design decisions from implementation, so one can change without breaking the other.

Tokens as contracts

Design decisions become stable, versionable interfaces between design and code.

Framework independence

Consume tokens in Tailwind, Sass, CSS-in-JS, or plain CSS without coupling to a specific stack.

Shared design language

Designers name it color-brand-primary. Developers use --color-brand-primary. Same thing, always in sync.

ARCHITECTURE

Three independent, decoupled layers.

Update your token values without touching components. Swap your component library without redefining your tokens.
Each layer owns its responsibility — nothing else.

Layer 01

Tokens

The only place where design decisions are defined. Change here, everything downstream follows.

Layer 02

Base styles

Apply tokens to native HTML elements, establishing predictable defaults and consistent rendering across every context.

Layer 03

Components

Reference patterns, not mandates. Use them as a starting point or ignore them entirely — the token layer works either way.

IMPLEMENTATION

Works with your stack. Any stack.

Install the package once. From there, consume tokens exactly the way your stack expects —
no adapters, no plugins, no build-time magic required.

CSS Variables

Plain CSS

Import the token file and use CSS custom properties directly. Works in any project — no framework needed.

# Install
npm install @mikaelcarrara/underlith

# Import
@import "underlith/src/underlith.tokens.css";

# Use
color: var(--ul-color-text);
gap: var(--ul-space-4);
border-radius: var(--ul-radius-md);

View all tokens →

Tailwind v4

Native @theme

Tailwind v4 reads CSS variables natively via @theme. Import Underlith tokens and Tailwind utilities consume them automatically — no config file needed.

# Import in your main CSS
@import "underlith/src/underlith.tokens.css";

@theme {
  --color-brand: var(--ul-color-brand-primary);
  --radius-md: var(--ul-radius-md);
}

# Use Tailwind classes normally
<button className="bg-brand text-white" />

Sass / Less

Variables & Mixins

Reference Underlith tokens inside Sass variables or mixins. Adopt without rewriting a single existing rule.

# Import
@import "underlith/src/underlith.tokens.css";

# Map to Sass variable
$color-text: var(--ul-color-text);

# Before → After
color: $old-text-color;
@mixin button-primary {
  background: $old-text-color;
}

TWO PATHS

The infrastructure for your
own source of truth.

Underlith doesn't replace your design decisions — it gives you the infrastructure to own them.
Whether you're starting from scratch or migrating an existing system, the destination is the same:
one package where your brand lives, versioned and shareable across every product.

New project

Two commands. Done.

Define your brand tokens, scaffold the package, publish. Every product your org builds from here installs @your-org/tokens and inherits your design decisions automatically.

$ underlith brand init → $ npm publish
# Scaffold your brand package
underlith brand init --org acme

# Publish to npm
cd @acme/tokens && npm publish --access public

# Any project installs it
npm install @acme/tokens
@import "@acme/tokens/acme.brand.css";
1

Run brand init: Define your brand values. The command scaffolds a ready-to-publish npm package.

2

Publish once: @your-org/tokens becomes the shared source of truth across every product your org builds.

Existing project

Three commands. Zero rewrites.

Already have a design system? Map your existing variables to Underlith tokens, extract your brand layer into a standalone package, publish. Components keep working exactly as before.

$ underlith init --shadcn → $ underlith brand init → $ npm publish
# 1. Map existing variables to Underlith tokens
underlith init --shadcn \
  --globals ./styles/globals.css

# Before → After (zero component changes)
--primary: var(--ul-primary);

# 2. Extract brand layer and scaffold package
underlith brand init --org acme

# 3. Publish
npm publish --access public
1

Integrate: Existing variables are mapped to Underlith tokens. No component rewrites.

2

Extract: Brand values are separated from framework primitives and scaffolded into @your-org/tokens.

3

Publish: Every product in your org installs the package. Change once, update everywhere.
