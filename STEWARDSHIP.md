# Token Stewardship

This document defines how Underlith tokens are owned, maintained, and evolved over time.

Stewardship is not about rules — it's about responsibility. Rules are defined in [GOVERNANCE.md](./GOVERNANCE.md). Stewardship is what happens between the rules: the ongoing care that keeps a token system healthy, coherent, and trustworthy as products and teams evolve.

---

## What is a token steward?

A token steward is the person or team responsible for the long-term health of a token category. Not just approving changes — but actively monitoring usage, identifying drift, proposing improvements, and ensuring the token vocabulary stays meaningful over time.

Every token category has a steward:

| Category | Steward responsibility |
|----------|----------------------|
| Colors | Palette coherence, semantic alias accuracy, brand alignment |
| Typography | Scale consistency, font decision traceability |
| Spacing | Scale integrity, no arbitrary values creeping in |
| Motion | Accessibility compliance, composite token coverage |
| Semantic aliases | Intent accuracy — does the name still match the meaning? |
| Brand layer (`@your-org/tokens`) | Package versioning, downstream impact awareness |

---

## Token lifecycle

Every token goes through a lifecycle. Stewards are responsible for managing each stage.

```
Proposal → Review → Active → Mature → Deprecated → Removed
```

**Proposal** — A new design decision needs a token. The steward evaluates whether it's truly a new decision or an existing token used differently.

**Review** — Schema validated, naming checked, semantic intent confirmed. Does this token name still make sense in 2 years?

**Active** — Token is in use. Steward monitors adoption via `underlith audit` coverage reports.

**Mature** — Token has stable usage across multiple products. Changes at this stage require major version bump.

**Deprecated** — Token is marked deprecated. Replacement is documented. Compatibility maintained for at least two releases.

**Removed** — Only after a major version bump. Never silently.

## Semantic versioning for design tokens

Underlith follows semver strictly. Stewards are responsible for classifying changes correctly.

| Change type | Version bump | Example |
|-------------|-------------|---------|
| New token added | patch | Adding `--ul-space-6` |
| Token value updated (non-breaking intent) | minor | Adjusting `--ul-duration-fast` from 150ms to 120ms |
| Token renamed | major | `--ul-color-brand` → `--ul-color-brand-primary` |
| Token removed | major | Removing a deprecated token |
| Semantic meaning changed | major | `--ul-color-surface` now maps to a different palette value |

**The steward's job is to catch major-bump changes before they ship** — not after.

When in doubt, bump major. Downstream products can always update. A silent breaking change erodes trust permanently.

## Token health indicators

A healthy token system has:

- **High coverage** — `underlith audit` shows 85%+ token adoption across consumer projects
- **Low duplication** — no two tokens express the same decision with different names
- **Stable aliases** — semantic token names haven't changed meaning without a major version
- **No orphans** — every token in the package is actually consumed by at least one product
- **Clean deprecation trail** — deprecated tokens have documented replacements and a removal timeline

Signs of an unhealthy system:

- Hardcoded values reappearing in PRs after migration
- Multiple tokens with overlapping semantic meaning
- Aliases that no longer reflect their intent (e.g. `--ul-color-text-primary` used for backgrounds)
- Tokens added without steward review accumulating over time

## Evolving the token vocabulary

Products evolve. Brand decisions change. New platforms emerge. The token vocabulary must evolve too — without breaking what depends on it.

**How to evolve responsibly:**

1. **Additive first** — add new tokens before deprecating old ones. Give consumers time to migrate.
2. **Never rename silently** — a rename is a breaking change. Always provide an alias period.
3. **Review semantic aliases quarterly** — does `--ul-color-text` still mean what it meant 6 months ago? Does the underlying value still reflect the intent?
4. **Audit before major releases** — run `underlith audit` across all consumer projects before publishing a major version. Know the impact before it ships.
5. **Document the why** — token changes without rationale become archaeological mysteries. Every change should have a reason in the changelog.

## Brand package stewardship (`@your-org/tokens`)

The `@your-org/tokens` package generated by `underlith brand init` has its own stewardship responsibilities — separate from Underlith core.

The brand package steward is responsible for:

- **Version discipline** — bumping versions correctly when brand decisions change
- **Downstream awareness** — knowing which products consume the package and what a version bump affects
- **Publish cadence** — not publishing breaking changes without coordinating with consuming teams
- **Changelog maintenance** — every publish has a clear record of what changed and why

The brand package is your org's source of truth. Its steward is the person your org trusts to own that truth.

---

## The steward's mindset

A token is not just a variable. It's a promise.

When a developer writes `var(--ul-color-brand-primary)`, they're trusting that this token will always mean "the primary brand color" — regardless of what framework they're using, what product they're building, or what year it is.

The steward's job is to keep that promise.

Not by resisting change — but by ensuring that when things change, they change with intention, with communication, and with respect for everything that depends on them.
