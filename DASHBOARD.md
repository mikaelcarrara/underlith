# Underlith Cloud Dashboard — MVP Specification

This document describes the MVP for the Underlith Cloud Dashboard: an observability and governance panel for the versioned token ecosystem generated by Underlith. The focus is visibility — not token editing.

## Goal

- Answer, simply and reliably:
-  1) Which projects are integrated with Underlith?
-  2) Which token package version is each project using?
-  3) How do tokens relate to projects (dependencies/coverage)?

## Non‑Goals (MVP)

- Token editing, push/publish, or Figma sync.
- User management and RBAC beyond the minimum for access.
- Deep compliance audits (only basic signalling via versions/usage).

---

## Dashboard Views (MVP)

1. Projects
   - KPIs: total projects, up‑to‑date, outdated, latest version.
   - Table: Project | Repository | Framework | Token package | Token version | Last sync | Status.
   - Status:
     - Up‑to‑date: version == latest.
     - Outdated: version < latest.
   - Purpose: understand adoption and maturity of the infrastructure.

2. Token Package
   - Metadata: name, latest, publish date, number of consuming projects, token count.
   - Token list grouped by category (colors, spacing, typography, radius).
   - Click a token → opens dependency detail (who uses it).

3. Version Adoption
   - List projects running older versions vs latest.
   - Purpose: prioritize upgrades.

4. Token Dependency Graph
   - Table view: Token | Used in (# projects).
   - Token detail: list of projects where the token appears.
   - Purpose: understand the change “blast radius”.

## CLI Integration (Ingestion)

Events that make a project appear/update in the registry:

- `underlith init`
- `underlith integrate`
- `underlith brand init`

The CLI sends an ingestion payload to the control plane:

```json
{
  "project": {
    "name": "web-app",
    "repo": "https://github.com/acme/web-app",
    "framework": "Next.js"
  },
  "tokens": {
    "package": "@acme/tokens",
    "version": "2.3.0"
  },
  "timestamp": "2026-03-09T10:00:00Z",
  "signature": "HMAC(base64)"
}
```

Requirements:

- HMAC signature (short‑lived service key; rotatable). Header `X-UL-Signature`.
- Idempotency via `project.name + timestamp` or nonce (deduplicate replays).
- Minimize scopes: dashboard side is read‑only; write only on the ingestion endpoint.

---

## API (Near‑term, for production)

- `POST /api/ingest` (protected)
  - Receives CLI payload, validates signature, persists project/version/timestamp.

- `GET /api/dashboard` (authenticated)
  - Returns the aggregate required by all MVP views.

- `GET /api/tokens/:token/usage` (authenticated)
  - Lists projects using the token.

Response shape (equivalent to the current mock):

```json
{
  "package": {
    "name": "@acme/tokens",
    "latestVersion": "2.3.0",
    "publishDate": "2026-03-01",
    "tokenCount": 126,
    "categories": {
      "colors": ["--ul-color-primary"],
      "spacing": ["--ul-space-md"],
      "typography": ["--ul-font-size-md"],
      "radius": ["--ul-radius-sm"]
    }
  },
  "projects": [
    {
      "name": "web-app",
      "repo": "https://github.com/acme/web-app",
      "framework": "Next.js",
      "tokenPackage": "@acme/tokens",
      "tokenVersion": "2.3.0",
      "lastSync": "2026-03-09T10:00:00Z"
    }
  ],
  "dependencies": {
    "--ul-space-md": ["web-app", "mobile-app"],
    "--ul-color-primary": ["web-app"]
  }
}
```

## Data Model (MVP)

- `Package`
  - `name` (string) — e.g., `@acme/tokens`
  - `latestVersion` (semver string)
  - `publishDate` (ISO)
  - `tokenCount` (int)
  - `categories` (map<string, string[]>)

- `Project`
  - `name` (string)
  - `repo` (url)
  - `framework` (string)
  - `tokenPackage` (string)
  - `tokenVersion` (semver string)
  - `lastSync` (ISO)

- `DependencyIndex`
  - map `tokenName` → `string[]` of projects.

## Authentication and Security (Phases)

1. Phase 0 — No auth (file://)
   - Local fallback for demo.

2. Phase 1 — Hosting gate
   - Simple protection via platform (Cloudflare Access, password at CDN).

3. Phase 2 — Corporate OAuth (recommended)
   - GitHub OAuth/OIDC with organization membership verification.
   - Minimal scopes (read:org).
   - CLI → signed `POST /api/ingest` (Service Token/HMAC).

4. Phase 3 — RBAC
   - Profiles: `viewer` (read) and `admin` (control plane management).
   - Access audit and CLI key rotation.

Additional considerations:

- No secrets or sensitive logs collected in the front‑end.
- Limit CORS to internal domains.

## Essential Telemetry (Later)

- Ingestion events received (count/success/errors).
- Project counts by status (up‑to‑date/outdated).
- Top tokens by usage (top‑N).

No PII; operational metrics only.

## Glossary

- **Token package**: versioned package (e.g., `@acme/tokens`) published by the Underlith pipeline.
- **Latest**: most recent package version known by the control plane.
- **Outdated**: project consuming a version lower than `latest`.
- **DependencyIndex**: index mapping tokens → projects where they appear (basis for the “blast radius”).
