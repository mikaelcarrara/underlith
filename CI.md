# CI & Automation

This document defines how CI, automation, and AI-assisted workflows interact with Underlith and the brand packages built on top of it.

Underlith is designed to operate as a stable, machine-readable design layer that enables deterministic CI validation, safe automation, and AI-assisted UI generation and refactoring.

Tokens act as hard constraints, not suggestions.

---

## Role of CI in Underlith

CI enforces governance rules automatically across two levels:

**Level 1 — Underlith core** (`@mikaelcarrara/underlith`)
Validates that base token changes don't break contracts for downstream consumers.

**Level 2 — Brand packages** (`@your-org/tokens`)
Validates that brand token changes are intentional, versioned, and don't cause visual regressions across products.

CI is not optional and cannot be bypassed.

---

## Core principles

- **Tokens are contracts** between design and code
- **Automation must be deterministic**
- **AI operates within constraints**
- **All outputs must be auditable**
- **Human approval remains mandatory for critical changes**

## Canonical inputs and outputs

### Inputs
- `src/tokens/**` — base token definitions (Underlith core)
- `your-org.brand.css` — brand token definitions (`@your-org/tokens`)
- `underlith.tokens.css` — full token file, input for `underlith brand init`
- Source code from consumer projects (for lint validation)

### Outputs
- `underlith.tokens.css` — all tokens as CSS Custom Properties
- `underlith.css` — tokens + base styles, main entry point
- `your-org.brand.css` — brand layer, generated by `underlith brand init`
- `your-org.base.css` — base layer, generated by `underlith brand init`
- `underlith.tokens.json` — token data for tools and agents *(roadmap)*
- CI reports (lint, breaking changes, audits)

## Required CI jobs

### 1. build:tokens

**Purpose**
Generate all distributable artifacts from the canonical token source.

**Responsibilities**
- Read canonical token definitions
- Generate `underlith.tokens.css` and `underlith.css`
- Ensure deterministic output

**Failure conditions**
- Invalid token schema
- Non-deterministic generation
- Missing artifacts

### 2. lint:tokens

**Purpose**
Validate the structural and semantic correctness of tokens.

**Validations**
- Schema compliance
- Naming conventions (`--ul-*` prefix)
- Alias resolution
- Deprecated token usage
- Motion policy compliance (`--ul-ease-*`, `--ul-duration-*`)

**Failure conditions**
- Invalid schema
- Broken references
- Missing `--ul-` prefix
- Illegal value changes

### 3. breaking-change detection

**Purpose**
Detect changes that break the token contract.

**Detected as breaking**
- Token removal
- Token rename
- Token value change without semver bump
- Semantic meaning change

**Behavior**
- CI fails by default
- Requires explicit approval from category owner
- Requires semver major bump

### 4. lint:code (token-aware)

**Purpose**
Prevent hardcoded design values from entering codebases.

**Checks**
- Detect raw colors, spacing, font sizes, radii
- Compare against allowed `--ul-*` token whitelist
- Report violations with suggested token replacements
- Detect raw durations or cubic-bezier values — suggest `--ul-duration-*` and `--ul-ease-*`

**Use cases**
- PR validation
- Automated design QA
- Large-scale token migrations

### 5. visual-regression (recommended)

**Purpose**
Ensure that token refactors do not unintentionally change visuals.

**Behavior**
- Compare before/after snapshots
- Validate visual equivalence
- Required for large-scale token migrations and brand package updates

### 6. audit-log

**Purpose**
Ensure traceability and compliance.

**Logs**
- Token generation
- `underlith brand init` executions
- `underlith init --shadcn` executions
- AI-assisted changes
- Versioned outputs

## Example CI job

```yaml
name: Tokens CI

on:
  pull_request:
    paths:
      - 'src/tokens/**'
      - 'package.json'

jobs:
  build-tokens:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v3
      - run: npm ci
      - run: npm run build:tokens
      - run: npm run lint:tokens

  breaking-changes:
    runs-on: ubuntu-latest
    needs: build-tokens
    steps:
      - uses: actions/checkout@v3
      - run: npm run detect:breaking

  validate-code:
    runs-on: ubuntu-latest
    needs: build-tokens
    steps:
      - uses: actions/checkout@v3
      - run: npm run lint:tokens-check

  visual-regression:
    runs-on: ubuntu-latest
    needs: breaking-changes
    if: needs.breaking-changes.result == 'success'
    steps:
      - uses: actions/checkout@v3
      - run: npm run visual-regression
```

## AI-assisted workflows

Underlith is designed to be safely consumed by AI and automation systems.

Tokens are explicit, semantic, framework-agnostic, and machine-readable. This allows AI to operate within a bounded design space — executing within constraints, not making design decisions.

### Real-world examples already in production

**`underlith init --shadcn`**
AI-assisted script that reads an existing `globals.css`, maps every CSS variable to an Underlith semantic token, and rewrites the file. Zero component changes. Deterministic output constrained entirely by the token contract.

**`underlith brand init`**
Reads `underlith.tokens.css`, separates brand tokens from base tokens using org name and keyword heuristics, and scaffolds a publishable `@your-org/tokens` package. The AI operates within the token structure — it doesn't invent values, it extracts and organizes them.

**`underlith migrate` *(roadmap)***
AI-assisted migration from legacy hardcoded values to tokens. Scans a codebase, identifies values that map to existing tokens, and replaces them deterministically.

## Token-driven AI prompts (examples)

### Component generation

```
Generate a Card component using only the following design tokens:

- Colors: --ul-color-neutral-0, --ul-color-neutral-200, --ul-color-neutral-900
- Spacing: --ul-space-3, --ul-space-4, --ul-space-5
- Radius: --ul-radius-md
- Typography: --ul-font-size-md, --ul-font-weight-medium

Do not introduce new values.
Output plain HTML and CSS.
```

### Token validation

```
Analyze the following CSS and report any values not mapped to --ul-* tokens.

Allowed:
- Spacing: --ul-space-1 through --ul-space-6
- Colors: --ul-color-neutral-* and --ul-color-brand-*
- Font sizes: --ul-font-size-sm, --ul-font-size-md, --ul-font-size-lg
```

### Cross-framework migration

```
Refactor this React component to plain HTML and CSS.

Replace all hardcoded values with --ul-* design tokens.
Preserve visual output exactly.
Do not introduce framework-specific patterns.
```

## How CI, AI and Governance connect

- Governance defines the rules
- CI enforces them
- AI operates within them

```
Design intent → Tokens → @your-org/tokens → CI constraints → AI / Automation → Validated output
```

AI is a constrained executor, not a decision-maker.
